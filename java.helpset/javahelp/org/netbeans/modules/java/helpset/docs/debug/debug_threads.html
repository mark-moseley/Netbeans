Debugging Threads in the IDE

See Also

All the threads created in the current session are displayed in the Threads window. You can open the Threads window by choosing Window > Debugging > Threads (Alt-Shift-7).

Changing the Current Thread

Only one thread is the current thread at any given time. By default, the current thread is the thread within the current session from which the debugger gained control. When you switch between threads to debug, the Call Stack and Local Variables windows are automatically updated to reflect the data applicable to that thread.

To change the current thread:

  • Double-click any thread in the Threads window to make it the current thread.

Suspending and Resuming Threads

You can suspend execution of a thread if you think it is causing problems and then later resume the thread once the problem is solved.

In the Threads window, the icon to the left of the thread name indicates whether the thread is suspended or waiting to be resumed.

Icon Description
Indicates the current thread
Indicates a thread that is running and is not current
Indicates a thread that is suspended and is not current

You can display the Suspend property as a boolean flag (True or False) in the Threads window by clicking the columns button () and select the Suspend checkbox.

  • The state of the Thread is independent of whether the thread is suspended or waiting to be resumed. A thread's state may be Running while the Suspend property is True.

To suspend or resume a thread:

  • Right-click the thread in the Threads window and choose the Resume or Suspend command.

Multi-threaded Applications

When debugging a multi-threaded application, all threads are suspended when a breakpoint is hit (unless the breakpoint settings specifically are set to not suspend all threads). When you then proceed to step through the application, all threads are resumed. This behavior is desirable in some cases but not in others.

For example, when debugging a multi-threaded application, a step in one thread (thread 1) may be interrupted by a breakpoint encountered in another thread (thread 2). You can control the threads that trigger breakpoints during the debugging session in the Threads window.

  1. Start your debugging session.
  2. When the debugger encounters a breakpoint in thread 1, all threads in the program are suspended and the program counter indicates the current breakpoint.
  3. Invoke a step in thread 1 to resume all threads.
    If a breakpoint is encountered in another thread thread 2 before the step in thread 1 is complete, all threads are suspended and the program counter indicates the current breakpoint in thread 2.

    If you want to ignore breakpoints not on thread 1, do one of the following:

    • In the Threads window, click Ignore Breakpoint. Clicking this will cause the debugger to ignore the specified breakpoint encountered in thread 2 and you can continue to step through the program following thread 1.
    • In the Threads window, click Ignore All Such Breakpoints. Clicking this will cause the debugger to ignore all breakpoints that do not occur on thread 1 until the end of the debugging session.
  4. Continue to step through your program.

Viewing Source Code for a Thread

You can view the source code of a thread in the IDE's Source Editor if you think it is causing problems and want to examine it further.

To view a thread's source:

  • Right-click the thread in the Threads window and choose Go To Source.

    If the source of the thread is available, the Source Editor jumps to the current call on the thread's call stack.
